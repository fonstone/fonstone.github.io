## 1. 概述

Claude Code 是 Anthropic 推出的终端编码代理，能够读取文件、执行命令、编辑代码并迭代完成任务。但它本身是通用型工具，社区围绕它发展出了多种框架，各自解决不同的问题。

本文对比三个主流框架：**Superpowers**、**GStack** 和 **GSD**，从各自的侧重点、优缺点和适用场景进行分析。

## 2. Superpowers：能力扩展层

### 2.1 核心理念

Superpowers 是一个增强层，专注于扩展 Claude Code 的**实际能力边界**。它的模式类似于浏览器扩展——Claude Code 正常运行，但 Superpowers 为其添加了额外的工具调用能力。

### 2.2 主要特性

- **扩展工具访问**：网络搜索、外部 API 调用、数据库查询等，作为 Claude Code 的原生工具使用
- **自定义斜杠命令**：定义自己的命令，Claude 可以用自然语言触发执行
- **增强的 CLAUDE.md 配置**：结构化的模板和模式，编写更有效的上下文文件
- **集成钩子**：将 Claude Code 会话连接到外部服务和工作流
- **会话记忆模式**：跨会话的持久化上下文

### 2.3 优点

- 直接解决 Claude Code 的默认限制（不知道网上有什么、无法查询数据库、无法触发 CI/CD）
- 插件式架构，按需启用，不改变 Claude Code 的核心使用方式
- 特别适合日常开发工作流，让 Claude 能与实际开发环境交互
- 跨会话记忆让长期项目不再丢失上下文

### 2.4 缺点

- 需要大量初始配置：连接工具、编写自定义命令、配置集成
- 插件模型意味着开发者需要自行维护这些集成，外部服务变更时需要手动更新
- 配置复杂度随工具数量线性增长

### 2.5 适用场景

- 个人开发者或小团队，希望将 Claude Code 扩展到完整工作流
- 重度依赖外部服务（API、数据库、监控工具）的开发者
- 已经触及默认 Claude Code 能力边界的用户

## 3. GStack：结构化优先

### 3.1 核心理念

GStack 是一个脚手架和项目管理框架，核心思想是：**当 Claude 在明确定义的技术上下文中工作时，表现更好**。它通过预定义技术栈、文件结构和约定来显式提供上下文。

### 3.2 主要特性

- **技术栈配置文件**：声明式定义技术栈，Claude 在行动前读取
- **项目脚手架**：生成与你的技术栈对齐的项目结构、样板代码和文件夹约定
- **约定执行**：Claude 在定义的约定内操作，而不是自由发挥
- **多文件感知**：跨多个文件处理上下文，不会丢失文件间的关联关系

### 3.3 优点

- 新项目启动时效果最佳，减少架构漂移
- 团队中多个开发者使用 Claude Code 时，行为保持一致
- 共享的技术栈配置意味着所有人使用相同的模式和约定
- 维护负担低，一旦配置好几乎不需要额外工作

### 3.4 缺点

- 结构既是优势也是约束：已有项目如果不符合干净的技术栈定义，接入成本高
- 假设你已经确定要用的技术栈，对探索性或研究型工作来说，前置配置是负担
- 不适合已有代码库的快速改造

### 3.5 适用场景

- 使用明确技术栈启动新项目的团队
- 希望 Claude 在整个项目中保持一致行为的开发者
- 曾因 Claude 在不同会话中引入架构不一致而困扰的团队

## 4. GSD：分阶段执行

### 4.1 核心理念

GSD（Get Stuff Done）是一个结构化的提示工程方法，将复杂编码任务分为三个独立阶段：**计划（Plan）→ 执行（Execute）→ 审查（Review）**。每个阶段获得干净的上下文窗口和清晰的目标。

### 4.2 三阶段详解

**计划阶段（Plan）**：只产出计划，不写代码。明确描述要构建或修改的内容、受影响的文件/模块/系统、约束条件。输出为结构化的实施计划。

**执行阶段（Execute）**：在全新会话中，根据计划编写代码。不要求 Claude 思考问题，只要求它执行计划。大型计划可拆分为多个子阶段。

**审查阶段（Review）**：再次开启新会话，将生成的代码与原始需求对比，检查正确性、边界情况、安全性、性能和代码质量。

### 4.3 优点

- 解决"上下文污染"问题：混合计划、编码和审查到一个会话中会降低输出质量
- 每个认知任务获得独立的干净工作空间，Claude 的注意力不会被无关上下文分散
- 与高级工程师分解工作的思维方式一致，是可重复的系统
- 减少调试和修正周期，总体效率更高

### 4.4 缺点

- 短期来看增加了前置结构时间，对于简单任务可能显得过度
- 需要严格的阶段纪律：忘记 `/clear` 或携带对话历史会破坏框架效果
- 自主性低，需要人工在阶段间传递结构化产物
- 不适合需要快速迭代的探索性编码

### 4.5 适用场景

- 复杂、多步骤的编码任务，需要高质量输出
- 团队希望标准化编码流程，确保每个任务都经过计划、执行、审查
- 曾因上下文污染导致 Claude 输出质量下降的开发者

## 5. 横向对比

| 维度 | Superpowers | GStack | GSD |
|------|-------------|--------|-----|
| **核心侧重点** | 能力扩展 | 项目结构 | 分阶段执行 |
| **解决的问题** | Claude 能做什么 | Claude 在什么上下文中工作 | Claude 如何组织工作 |
| **自主性级别** | 中 | 低 | 低 |
| **配置复杂度** | 中-高 | 中 | 低 |

| **团队 vs 个人** | 个人/小团队 | 团队 | 个人/团队 |
| **最佳工作流** | 交互式开发 | 绿地开发 | 交互式开发 |

## 6. 如何选择

### 选 Superpowers 如果：

- 你希望 Claude Code 能做更多事——连接数据库、搜索网络、触发外部工作流
- 你主要独立开发或在小团队中工作
- 你愿意在前置配置上投入时间以换取后续效率
- 你的工作需要 Claude 与本地代码库之外的系统交互

### 选 GStack 如果：

- 你正在启动新项目且有明确的技术栈
- 你的团队需要一致的 Claude 行为
- 你曾遇到 Claude 在不同会话中引入不一致的问题
- 架构纪律对你来说比原始能力更重要

### 选 GSD 如果：

- 你经常处理复杂的多步骤编码任务
- 你发现 Claude 在长会话中输出质量逐渐下降
- 你希望每个任务都经过计划、执行、审查的完整流程
- 你愿意用少量前置时间换取更少的调试和修正

## 7. 组合使用

这三个框架并非互斥，它们解决的是不同层面的问题：

- **Superpowers + GStack**：最自然的组合。GStack 提供项目结构，Superpowers 提供工具扩展，两者互补。
- **GSD + Superpowers**：GSD 提供阶段纪律，Superpowers 为每个阶段提供更强的工具能力。
- **三者组合**：GStack 定义项目结构，Superpowers 扩展工具能力，GSD 组织工作流程。适合大型团队和复杂项目。

## 8. 总结

| 框架 | 一句话总结 |
|------|-----------|
| **Superpowers** | 解决"能力"问题——让 Claude Code 能做更多事 |
| **GStack** | 解决"结构"问题——让 Claude Code 在正确的上下文中工作 |
| **GSD** | 解决"流程"问题——让 Claude Code 以正确的方式组织工作 |

Claude Code 框架生态仍在快速发展中。更持久的决策原则是：先明确你真正要解决的问题是**结构**、**能力**还是**流程**，然后再做选择。
