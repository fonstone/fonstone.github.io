## 1. 概述

OpenSpec 是一个 **AI-Native 的规范驱动开发系统（Spec-Driven Development, SDD）**，为 AI 编码助手提供轻量级规范层。核心理念是：在编写第一行代码之前，先与 AI 对齐需求，让开发过程更可预测、更高效。

- **项目地址**：[github.com/Fission-AI/OpenSpec](https://github.com/Fission-AI/OpenSpec)
- **中文文档**：[openspec.radebit.com](https://openspec.radebit.com)
- **开源协议**：MIT
- **运行环境**：Node.js 20.19.0+
- **Stars**：42.7k+

## 2. 设计哲学

OpenSpec 围绕四条核心原则构建：

| 原则 | 含义 |
|------|------|
| **流动而非僵化** | 没有阶段门控，按需处理任何产物 |
| **迭代而非瀑布** | 随着开发深入不断学习和精化 |
| **简单而非复杂** | 轻量级设置，最少的仪式感 |
| **存量项目优先** | 适配现有代码库，不只是绿地项目 |

与传统规范系统不同，OpenSpec 不强制你在"计划→实施→完成"的阶段里依次推进。你可以在任何时间点更新任何产物，凭自己的节奏推进工作。

### 为什么需要 OpenSpec？

AI 编码助手功能强大，但当需求只存在于聊天记录中时，结果往往难以预测。OpenSpec 在人与 AI 之间建立了一层结构化的共识层：

- **先对齐再编码** — 避免方向性偏差带来的大量返工
- **井然有序** — 每个变更都有独立文件夹，包含提案、规范、设计和任务
- **流动灵活** — 随时更新任意产物，无需遵守僵化流程
- **工具生态广泛** — 支持 25+ AI 助手（Claude Code、Cursor、Windsurf 等）

### 与同类产品对比

| 对比对象 | 特点 |
|---------|------|
| vs. **Spec Kit** (GitHub) | Spec Kit 全面但重量级，有严格阶段门控、大量 Markdown 文件、Python 环境依赖。OpenSpec 更轻量，让你自由迭代 |
| vs. **Kiro** (AWS) | Kiro 功能强大但锁定在其 IDE 内，且仅限 Claude 模型。OpenSpec 支持你现有的全部工具 |
| vs. **不用规范** | 没有规范的 AI 编码意味着模糊提示和不可预测的结果。OpenSpec 在不增加繁文缛节的前提下带来可预测性 |

## 3. 基本原理

### 3.1 整体架构

OpenSpec 将你的工作组织为两大核心区域：

<MermaidDiagram chart={`graph LR
    subgraph openspec["openspec/"]
        specs["specs/<br/>权威基准<br/>系统当前行为描述"]
        changes["changes/<br/>提议的修改<br/>每个变更 = 一个文件夹<br/>包含产物 + Delta 规范"]
    end
    changes -- "归档时合并" --> specs
`} />

- **Specs（规范）** — 系统当前行为的权威基准（Source of Truth），描述"系统现在怎么运作"
- **Changes（变更）** — 提议的修改，各自为独立文件夹，完成后归档合并回 Specs

这种分离使得你可以同时进行多个变更而不产生冲突，归档时再干净地合并。

### 3.2 Specs（规范）

Specs 描述系统行为，使用结构化的需求和场景：

```
openspec/specs/
├── auth/
│   └── spec.md           # 认证行为
├── payments/
│   └── spec.md           # 支付处理
└── ui/
    └── spec.md           # UI 行为和主题
```

**规范格式包含以下要素：**

| 要素 | 作用 |
|------|------|
| `## Purpose` | 领域的高层次描述 |
| `### Requirement:` | 系统必须具备的特定行为 |
| `#### Scenario:` | 需求的具体示例 |
| SHALL/MUST/SHOULD | RFC 2119 关键词，表示需求强度 |

**核心原则：Spec 是行为契约，而非实现计划。** 好的规范描述可观测的行为（输入、输出、错误条件），避免包含内部类名、框架选择或逐步实现细节。

### 3.3 Changes（变更）

每个变更是一个独立文件夹，包含实施该修改所需的一切：

```
openspec/changes/add-dark-mode/
├── proposal.md           # 为什么做、做什么
├── design.md             # 技术实现方案
├── tasks.md              # 实现任务清单
├── .openspec.yaml        # 变更元数据（可选）
└── specs/                # Delta 规范
    └── ui/
        └── spec.md       # ui/spec.md 中要修改的内容
```

变更为文件夹的好处：
1. **一切汇聚** — 提案、设计、任务和规范都在一处
2. **并行工作** — 多个变更可同时存在而不冲突
3. **历史清晰** — 归档后保持完整上下文
4. **便于审查** — 打开文件夹即可了解全貌

### 3.4 Artifacts（产物）

产物是变更内引导工作的文档，按依赖关系层层推进：

<MermaidDiagram chart={`graph LR
    A["proposal<br/>why + scope"] --> B["specs<br/>what changes"]
    B --> C["design<br/>how / approach"]
    C --> D["tasks<br/>steps to take"]
    D --> E["implement"]
    E -. "更新产物" .-> A
    E -. "更新产物" .-> B
    E -. "更新产物" .-> C
`} />

#### 四种产物类型

| 产物 | 文件 | 用途 | 何时更新 |
|------|------|------|---------|
| **Proposal** | `proposal.md` | 捕获意图、范围和高层方案 | 范围变化、意图明确化、方案根本性转变时 |
| **Specs** | `specs/` | Delta 规范，描述相对当前规范的变更 | 需求变更时 |
| **Design** | `design.md` | 技术方案和架构决策 | 实现中发现方案需要调整时 |
| **Tasks** | `tasks.md` | 具体实现清单，带复选框 | 随开发推进更新任务状态 |

### 3.5 Delta 规范

Delta 规范是 OpenSpec 适配存量项目开发的核心概念。它描述**什么在变化**，而非重述整个规范：

```markdown
# Delta for Auth

## ADDED Requirements

### Requirement: Two-Factor Authentication
The system MUST support TOTP-based two-factor authentication.

#### Scenario: 2FA enrollment
- GIVEN a user without 2FA enabled
- WHEN the user enables 2FA in settings
- THEN a QR code is displayed for authenticator app setup

## MODIFIED Requirements

### Requirement: Session Expiration
The system MUST expire sessions after 15 minutes of inactivity.
(Previously: 30 minutes)

## REMOVED Requirements

### Requirement: Remember Me
(Deprecated in favor of 2FA)
```

| Delta 部分 | 含义 | 归档时操作 |
|------------|------|-----------|
| `ADDED` | 新增行为 | 追加到主规范 |
| `MODIFIED` | 修改行为 | 替换现有需求 |
| `REMOVED` | 废弃行为 | 从主规范删除 |

**Delta 优于全量规范的原因：**
- **清晰** — 一目了然看到什么在变
- **避免冲突** — 两个变更可触及同一规范文件而不冲突
- **审查高效** — 审阅者只需关注变化
- **适配存量** — 大多数工作是修改现有行为，Delta 让修改成为一等公民

### 3.6 Schemas（Schema 机制）

Schema 定义工作流中产物类型及其依赖关系：

```yaml
# openspec/schemas/spec-driven/schema.yaml
name: spec-driven
artifacts:
  - id: proposal
    generates: proposal.md
    requires: []              # 无依赖，可首先创建

  - id: specs
    generates: specs/**/*.md
    requires: [proposal]      # 需要先有提案

  - id: design
    generates: design.md
    requires: [proposal]      # 可与 specs 并行创建

  - id: tasks
    generates: tasks.md
    requires: [specs, design] # 需要 specs 和 design 都完成
```

依赖图：

<MermaidDiagram chart={`graph TD
    P["proposal<br/>（根节点）"]
    P --> S["specs<br/>（依赖 proposal）"]
    P --> D["design<br/>（依赖 proposal）"]
    S --> T["tasks<br/>（依赖 specs + design）"]
    D --> T
`} />

**依赖是使能器而非门控** — 它们展示创建顺序的可能性，而非强制必须按序创建。你可以跳过 design，也可以在 specs 之前或之后创建它。

### 3.7 归档机制

归档通过将 Delta 规范合并到主规范来完成变更：

<MermaidDiagram chart={`graph TB
    subgraph before["归档前"]
        direction TB
        spec_before["specs/auth/spec.md"]
        subgraph change["changes/add-2fa/"]

            t["tasks.md"]

    subgraph after["归档后"]

            d2["design.md"]
            t2["tasks.md"]

`} />

归档过程：ADDED 追加 → MODIFIED 替换 → REMOVED 删除 → 文件夹移至 `archive/`

`} />

## 4. 安装与初始化

### 4.1 安装

```bash
# 全局安装（需要 Node.js 20.19.0+）
npm install -g @fission-ai/openspec@latest

# 也支持 pnpm、yarn、bun、nix
```

### 4.2 初始化项目

```bash
cd your-project
openspec init
```

初始化后项目会新增如下结构：

```
openspec/
├── specs/              # 权威基准（系统当前行为）
│   └── <domain>/
│       └── spec.md
├── changes/            # 提议的变更（每个变更一个文件夹）
│   └── <change-name>/
│       ├── proposal.md
│       ├── design.md
│       ├── tasks.md
│       └── specs/      # Delta 规范（什么在变化）
│           └── <domain>/
│               └── spec.md
└── config.yaml         # 项目配置（可选）
```

## 5. 使用方式

### 5.1 核心工作流命令

#### `/opsx:propose` — 快速发起变更（核心模式默认）

一次性创建变更及其所有规划产物，这是最快的方式：

```
You: /opsx:propose add-dark-mode

AI:  已创建 openspec/changes/add-dark-mode/
     ✓ proposal.md — 为什么做、做什么
     ✓ specs/       — 需求与验收场景
     ✓ design.md    — 技术实现方案
     ✓ tasks.md     — 实现任务清单
     准备开始实现！
```

#### `/opsx:explore` — 探索问题空间

在确定变更方向之前，先探索和调查：

```
You: /opsx:explore

AI:  你想探索什么？

You: 我想改善页面加载性能，但不确定瓶颈在哪里。

AI:  让我来调查一下...
     [分析打包大小，识别慢查询，审查组件渲染模式]
     
     发现三个主要瓶颈：
     1. 大量未优化图片
     2. ProductList 中的同步数据获取
     3. Context 变更导致的重渲染
```

#### `/opsx:apply` — 实现任务

让 AI 开始实现 tasks.md 中的任务：

```
You: /opsx:apply

AI:  正在实现各项任务...
     ✓ 1.1 创建 ThemeContext
     ✓ 1.2 创建主题切换组件
     ✓ 2.1 添加 CSS 变量
     ✓ 2.2 接入 localStorage
     所有任务完成！
```

#### `/opsx:archive` — 归档变更

完成变更，将 Delta 规范合并到主规范：

```
You: /opsx:archive

AI:  正在归档 add-dark-mode...
     ✓ 已合并规范到 openspec/specs/ui/spec.md
     ✓ 已移至 openspec/changes/archive/2025-01-24-add-dark-mode/
     规范已更新。准备好下一个功能！
```

### 5.2 扩展工作流命令（需启用扩展模式）

| 命令 | 用途 | 何时使用 |
|------|------|---------|
| `/opsx:new` | 创建变更脚手架 | 显式控制产物创建步骤 |
| `/opsx:continue` | 创建下一个产物 | 逐步创建，边做边想 |
| `/opsx:ff` | 快速创建所有规划产物 | 范围明确，一步到位 |
| `/opsx:verify` | 验证实现与规范的一致性 | 归档前的质量检查 |
| `/opsx:sync` | 同步 Delta 规范到主规范 | 需要提前同步时 |
| `/opsx:bulk-archive` | 批量归档多个变更 | 并行工作完成后 |
| `/opsx:onboard` | 引导新成员了解项目 | 新人上手时 |

### 5.3 CLI 命令

```bash
# 初始化与更新
openspec init          # 在项目中初始化 OpenSpec
openspec update        # 更新 AI 指令，确保最新 Slash 命令激活

# 配置
openspec config profile  # 选择工作模式（core 或 expanded）

# 浏览
openspec list          # 列出活跃变更
openspec show <name>   # 查看变更详情
openspec view          # 交互式仪表盘

# 验证
openspec validate <name>  # 验证规范格式

# Schema
openspec schema init <name>      # 从零创建自定义 Schema
openspec schema fork <from> <name>  # 基于现有 Schema 创建
```

### 5.4 `/opsx:ff` vs `/opsx:continue` 如何选择？

| 场景 | 推荐命令 |
|------|---------|
| 需求明确，准备开始构建 | `/opsx:ff` |
| 正在探索，想审查每一步 | `/opsx:continue` |
| 想先迭代提案再写规范 | `/opsx:continue` |
| 时间紧迫，需要快速推进 | `/opsx:ff` |
| 复杂变更，需要精细控制 | `/opsx:continue` |

| 情况 | 操作 |
|------|------|
| 意图不变，执行微调 | 更新现有变更 |
| 范围缩小（先做 MVP） | 更新现有变更 |
| 实现过程中发现方案需要调整 | 更新现有变更 |
| 意图根本改变了 | 新建变更 |
| 范围膨胀到完全不同的工作 | 新建变更 |
| 原变更可以独立完成 | 先归档，再新建 |

## 7. 验证机制

`/opsx:verify` 从三个维度验证实现是否符合规范：

| 维度 | 检查内容 |
|------|---------|
| **完整性** | 所有任务是否完成、所有需求是否有对应代码、场景是否覆盖 |
| **正确性** | 实现是否匹配规范意图、边缘情况是否处理、错误状态是否一致 |
| **一致性** | 设计决策是否反映在代码结构中、命名约定是否与 design.md 一致 |

验证不会阻止归档，但会提醒你潜在问题。

## 8. 自定义配置

### 自定义 Schema

你可以为团队创建自定义工作流 Schema：

```bash
# 从零创建
openspec schema init research-first

# 或基于现有 Schema 分叉
openspec schema fork spec-driven research-first
```

示例自定义 Schema（研究优先工作流）：

```yaml
# openspec/schemas/research-first/schema.yaml
name: research-first
artifacts:
  - id: research
    generates: research.md
    requires: []

  - id: proposal
    generates: proposal.md
    requires: [research]

  - id: tasks
    generates: tasks.md
    requires: [proposal]
```

### 规范强度级别

OpenSpec 支持**渐进式严格**：

- **轻量规范（默认）**— 简短的行为优先需求、清晰的范围和非目标、少量验收检查
- **完整规范（高风险场景）**— 跨团队/跨仓库变更、API/契约变更、安全/隐私关注、模糊性可能导致昂贵返工的变更

大多数变更应保持轻量模式。

## 9. 最佳实践

1. **保持变更聚焦** — 一个逻辑工作单元一个变更。如果同时"加功能X"和"重构Y"，考虑分为两个变更
2. **需求不明确时先 explore** — `/opsx:explore` 帮助在创建产物前理清思路
3. **归档前先 verify** — `/opsx:verify` 帮你发现实现与规范之间的不一致
4. **命名要清晰** — 好的变更名字让 `openspec list` 真正有用：
   - 好：`add-dark-mode`、`fix-login-redirect`、`optimize-product-query`
   - 差：`feature-1`、`update`、`changes`、`wip`
5. **保持上下文清洁** — OpenSpec 在清洁的上下文窗口下效果最好，开始实现前清除上下文
6. **推荐使用高推理模型** — OpenSpec 在 Opus 4.5 和 GPT 5.2 等高推理模型上效果最佳

## 10. 完整示例：添加暗色模式

### Step 1：发起变更

## 11. 术语表

| 术语 | 定义 |
|------|------|
| **Artifact（产物）** | 变更内的文档（proposal、design、tasks 或 Delta specs） |
| **Archive（归档）** | 完成变更并将 Delta 合并到主规范的过程 |
| **Change（变更）** | 对系统的提议修改，打包为包含产物的文件夹 |
| **Delta Spec（Delta 规范）** | 描述相对当前规范的变更（ADDED/MODIFIED/REMOVED） |
| **Domain（领域）** | 规范的逻辑分组（如 `auth/`、`payments/`） |
| **Requirement（需求）** | 系统必须具备的特定行为 |
| **Scenario（场景）** | 需求的具体示例，通常使用 Given/When/Then 格式 |
| **Schema** | 产物类型及其依赖关系的定义 |
| **Spec（规范）** | 描述系统行为的规格文档，包含需求和场景 |
| **Source of Truth（权威基准）** | `openspec/specs/` 目录，包含当前公认的系统能行 |
